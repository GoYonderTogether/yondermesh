# yondermesh — guide for AI coding agents

**yondermesh** is a self-hosted Agent Context Bus: one daemon + one MCP server
that lets your AI coding agents (Claude Code, Codex, Aider, Gemini, Cursor,
Windsurf, Trae, Continue, …) see each other, query each other, and hand off
tasks across devices and CLIs. No UI, no cloud lock-in, no agent modification.

## This file is a map, not a manual

> Harness engineering principle: **give the agent a map, not a thousand-page handbook**.
> Architecture details, data models, and contracts live in the files this map points to —
> don't pile detail here (it goes stale, bloats context, and can't be mechanically verified).
> When you change the architecture, update `ARCHITECTURE.md`, not this file.

**Read these first (jump as needed):**

- **`README.md`** — How to run + the 30-second pitch + quickstart.
- **`ARCHITECTURE.md`** — Repository codemap: what every file/directory does, architectural invariants, module boundaries. "Where does X live?" — check here first.
- **`CONTRIBUTING.md`** — How to contribute; the doc-sync discipline lives here too.
- **`CHANGELOG.md`** — Version history. Every published `package.json` bump needs an entry.
- **`SECURITY.md`** — Reporting vulnerabilities; threat model summary.

## Conventions

- TypeScript throughout. Run `npm run typecheck` before committing.
- CLI is the primary user surface (`ymesh <command>`). All commands are listed in `src/bin/ymesh.ts`; the help text is the canonical command list.
- Adapters live under `src/<cli-name>/` and follow the importer/wrapper/inject/extractor pattern (see `ARCHITECTURE.md`).
- Tests run with vitest: `npm test`. Each adapter has its own `*.test.ts`.
- The public docs site is in `site/` (VitePress). Auto-generated pages live under `site/reference/` and are regenerated by `scripts/docs/sync-all.mjs`.

## Doc-sync discipline (highest priority: code change = doc change)

> Docs naturally lag behind code. This project treats doc/code sync as a **hard rule**:
> every change must update the corresponding docs in the **same commit**.
> **Doc lag = an unfinished bug.** Self-check with the table below before marking done,
> then run the `doc-sync` skill (canonical at `skills/doc-sync/SKILL.md`) for a full audit.

| You changed… | Must also update |
|---|---|
| `src/bin/ymesh.ts` (CLI commands / flags / help text) | `npm run sync --prefix site` regenerates `site/reference/cli.md` (en + zh) from `ymesh help`. Commit the result. |
| `src/<adapter>/` (added / removed / coverage level changed) | `npm run sync --prefix site` regenerates `site/reference/adapters.md` (en + zh) from `src/*/`. Commit the result. |
| Daemon protocol, MCP tool list, mount strategies | `site/reference/mcp-tools.md`, `site/guide/mount.md`, `ARCHITECTURE.md` codemap |
| `package.json` version bump | `CHANGELOG.md` entry in the same commit |
| New top-level feature | `README.md` feature list + new `site/guide/<topic>.md` page linked from `site/.vitepress/config.ts` sidebar |
| `~/.yondermesh/` file layout | `site/reference/files.md` |
| `install.sh` or release / update flow | `site/guide/installation.md` |
| Doc/code mismatch, or a TODO left behind | Open an issue; do not let it rot silently |

### Hygiene gates (run before "done")

```bash
# 1. regen + drift check
node scripts/docs/check-drift.mjs

# 2. internal link check
node scripts/docs/verify-links.mjs

# 3. full docs build
npm run build --prefix site
```

All three must pass.

## Code quality

- **Surgical diffs** — every line traces to the task. No drive-by cleanup.
- **Verify before "done"** — code complete ≠ task done. Run `npm test`, `npm run typecheck`, and exercise the real path (CLI / daemon / MCP). Post the evidence.
- **Fail loud** — every "done" lists what was skipped, what warned, what wasn't verified.

## Release discipline

- `package.json` `version` bump → `CHANGELOG.md` entry in the same commit.
- `npm publish` is manual (no automated publish workflow yet). Tag the release: `git tag v0.Y.Z && git push --tags`.
- After publish, the install script (`install.sh`) and `ymesh update` should pull the new version on fresh installs.

## Internal docs (gitignored)

`docs/` at the repo root is **gitignored** — it contains internal architecture
specs, implementation loops, and acceptance baselines that are not open-source.
Do not commit anything under `docs/`. Public docs go in `site/` or top-level
`*.md` (`README.md`, `ARCHITECTURE.md`, `CONTRIBUTING.md`, `CHANGELOG.md`,
`SECURITY.md`).
